# AdaptAI 剧本 YAML Schema 规格说明

---

## 一、概述

AdaptAI 将小说文本转换为结构化剧本，输出格式为 YAML。本 Schema 定义了剧本的完整数据结构，所有字段名均使用中文，匹配影视行业中文剧本写作习惯。

---

## 二、顶层结构

```yaml
格式版本: "1.0.0"

元数据:
  剧本标题: "示例剧本"
  原著作者: ""
  转换工具: "AI"
  使用模型: "deepseek-chat"
  转换日期: "2026-06-08"
  来源章节数: 3

角色表:
  - id: "c1"
    name: "林墨"
    aliases: ["老林"]
    description:
      physical: "中年男子，身形瘦削"
      personality: "沉默寡言，内心敏感"
      role: "protagonist"
    relationships:
      - target_id: "c2"
        type: "friend"
        note: "多年老友"

场景列表:
  - 场景编号: "s1"
    来源章节: 1
    场景标题:
      地点: "内景 林墨的客厅"
      时段: "早晨"
      场景细节: "晨光透过窗帘缝隙，房间里弥漫着咖啡的气味"
    出场角色: ["c1"]
    情绪基调: "宁静中带着一丝不安"
    动作列表:
      - 类型: "动作"
        内容: "林墨站在窗前，手指无意识地敲击窗框。"
      - 类型: "行为提示"
        内容: "他忽然转身，快步走向书桌。"
    对白:
      - 角色编号: "c1"
        情绪: "低沉"
        表演提示: "对着电话"
        台词内容: "我到了。"
        行为提示: "拿起电话"
    镜头建议:
      - 镜头类型: "特写"
        镜头描述: "特写林墨敲击窗框的手指"
        可选: true
    转场方式: null

改编说明:
  - 场景编号: "s1"
    分类: "视觉"
    严重程度: "建议"
    内容: "开场可用慢动作展现手指敲击的节奏感。"

统计信息:
  场景数量: 2
  角色数量: 2
  对白总数: 5
  预估时长分钟: 4.0
```

## 三、字段详解

### 3.1 角色表 (Character)

```yaml
角色表:
  - id: "c1"                   # string, 格式 "c"+数字, 全局唯一
    name: "林墨"                # string, 主要称呼
    aliases: ["老林"]           # list[string], 别称/绰号
    description:
      physical: "中年，瘦削"    # string, 外貌描写
      personality: "沉默寡言"   # string, 性格特征
      role: "protagonist"      # enum: protagonist|antagonist|supporting|minor|cameo
    relationships:
      - target_id: "c2"       # string, 目标角色 ID
        type: "friend"         # enum: family|friend|rival|romantic|mentor_student|colleague|enemy|other
        note: "多年老友"       # string, 关系说明
```

**设计原因**：

- **`id` 用 `c` 前缀**而非纯数字：与场景编号（`s1`）和地点编号（`l1`）肉眼可区分，降低编辑时的认知负荷
- **`aliases` 单独存储**：小说中同一角色常有多个称呼（"老林"、"墨叔"），AI 需要记录所有别名才能正确识别后续章节中的角色指代
- **`role` 使用枚举**：限制 AI 输出为 5 个标准类型，防止幻觉生成"主要反派"、"半主角"等非标分类
- **`relationships` 内嵌于角色**：关系数据属于角色而非独立表。这是刻意违反数据库范式的设计——原因是在 Map/Reduce 阶段需要将角色带关系一起注入 Prompt，独立关系表需要额外 JOIN 操作，增加上下文组装复杂度

### 3.2 场景列表 (Scene)

每个场景是一个独立的时间空间单元。当时间或地点发生显著变化时，即应划分为新场景。

#### 3.2.1 场景编号与来源

```yaml
场景编号: "s1"      # string, 格式 "s"+数字, 全局递增
来源章节: 1         # int, 来源章节编号
```

**设计原因**：`来源章节` 保留溯源能力。作者需要对照修改时，可以直接定位到原文章节。这是 AI 辅助改编"可解释性"的基础。

#### 3.2.2 场景标题 (Scene Heading)

```yaml
场景标题:
  地点: "内景 咖啡店"          # string, "内景/外景" + 具体地点
  时段: "早晨"                 # enum: 黎明|早晨|下午|傍晚|夜晚|稍后|连续
  场景细节: "店内播放着爵士乐"  # string, 环境氛围描述
```

**设计原因**：

- **`地点` 带内景/外景前缀**：直接决定拍摄方式——内景（INT.）需要棚拍，外景（EXT.）需要勘景。前端根据前缀渲染不同图标
- **`时段` 用 7 个中文枚举**：比好莱坞标准的 4 个（DAY/NIGHT/DAWN/DUSK）更细腻。中文小说中"下午"和"傍晚"的氛围差异很大，合并为同一个枚举会导致灯光方案不够精确。"稍后"和"连续"用于同一地点的连续时间跨越
- **`场景细节`**：静态环境描写，供美术指导和摄影指导参考。与动作列表的区别——这是布景，不是表演

#### 3.2.3 出场角色

```yaml
出场角色: ["c1", "c2"]   # list[string], 角色 ID 列表
```

**设计原因**：用角色 ID 而非角色名。这有两个目的：
1. **交叉校验**：Validator 检查所有出场角色是否存在于角色表中，ID 匹配比名称匹配更可靠（处理别名问题）
2. **通告单生成**：剧组每天发通告单时可以直接用 ID 反查角色全名和扮演者

#### 3.2.4 情绪基调

```yaml
情绪基调: "宁静中带着一丝不安"   # string, 1-2句氛围描述
```

**设计原因**：为导演和演员提供场景的情绪锚点。这不是好莱坞标准格式的要求，但中文剧本创作中"情绪基调"是常见惯例——它为整个场景的表演和灯光风格定调。

#### 3.2.5 动作列表 (Action Block)

```yaml
动作列表:
  - 类型: "动作"                     # enum: 动作 | 行为提示
    内容: "林墨走到窗前，推开窗户。"  # string, 动作描写文本
  - 类型: "行为提示"
    内容: "（突然想起什么事）"
```

**设计原因**：

- **采用对象数组而非纯字符串数组**：每个动作块除了内容还需要标注类型。`动作` 是纯视觉描写（摄影机可见的），`行为提示` 是内嵌于动作中的角色心理或微观动作提示（类似"（一惊）"、"（顿了顿）"）。分开标注让 AI 生成的剧本更接近中文编剧的写作习惯
- **是剧本的"肉"**：与小说原文的心理描写不同，动作块必须是纯视觉化的（Show, Don't Tell）
- **每个动作块对应一个"镜头意图"**：方便后续分镜工作。前端按动作块独立渲染

#### 3.2.6 对白 (Dialogue)

```yaml
对白:
  - 角色编号: "c1"          # string, 说话角色 ID
    情绪: "愤怒"             # string, 情绪状态
    表演提示: "压低声音"     # string, 表演提示
    台词内容: "你早就知道。"  # string, 对白原文
    行为提示: "握紧拳头"     # string, 说话时的行为
```

**设计原因**：

- **用 `角色编号` 而非角色名**：保持与出场角色的 ID 一致性，便于 Validator 做对白归属检查
- **`情绪` 和 `表演提示` 分离**：情绪是"这个角色现在是什么状态"（愤怒、悲伤），表演提示是"这个角色怎么演绎这句台词"（压低声音、对着窗外）。两个维度分开可以让导演和演员各自关注自己需要的部分
- **`行为提示` 独立于台词**：演员说台词时的伴生动作（握拳、踱步），属于表演指导而非台词本身
- **所有字符串字段均可空**：不是每句台词都需要所有标注。强制填写会导致 AI 生成"（平静地说）"这种无意义的占位文本

#### 3.2.7 镜头建议 (Shot Suggestion)

```yaml
镜头建议:
  - 镜头类型: "特写"             # enum: 全景|特写|中景|跟拍|横摇|俯仰|滑轨|航拍|主观视角|过肩|插入镜头|切出
    镜头描述: "特写他的手指"      # string, 具体镜头描述
    可选: true                   # bool, 始终为 true——AI 建议非强制
```

**设计原因**：

- **始终标记 `可选: true`**：镜头建议是 AI 的视觉化建议，最终决定权在导演。这个字段的存在是为了给 AI 生成的剧本增加"可拍摄性"，但不应该被认为是强制指令
- **12 种镜头类型枚举**：覆盖常用镜头语言，既给 AI 足够的选择空间，又防止幻觉输出不存在的镜头类型

#### 3.2.8 转场方式

```yaml
转场方式: "切至"   # enum: 切至|淡入|淡出|叠化至|跳切至|匹配切至 | null
```

**设计原因**：`null` 表示默认硬切（CUT TO）。AI 只在有特殊转场需求时才填写具体值，避免每个场景都生成"切至"（相当于什么都没说）。

### 3.3 改编说明 (Adaptation Note)

```yaml
改编说明:
  - 场景编号: "s1"           # string | null, 作用于特定场景
    来源章节: 1              # int | null, 作用于特定章节
    分类: "视觉"             # enum: 节奏|对白|结构|角色|视觉
    严重程度: "建议"          # enum: 建议|推荐|警告
    内容: "开场可用慢动作"    # string, 具体建议
```

**设计原因**：

- **`场景编号` 和 `来源章节` 均可为 null**：一条改编说明可以作用于特定场景、特定章节，或全局（两者均为 null）
- **三维分类**：`分类` 标注建议类型（节奏/对白等），`严重程度` 标注紧迫程度。这样前端可以按类型筛选或按严重程度排序展示
- **AI 改编决策的可追溯性**：这些备注记录了 AI 在改编过程中做出的关键决策（"为什么把这段心理描写删了？"——"改编说明里写了：无法视觉化"）

### 3.4 统计信息 (Stats)

```yaml
统计信息:
  场景数量: 2
  角色数量: 2
  对白总数: 5
  预估时长分钟: 4.0    # 按每场景约 2 分钟估算
```

**设计原因**：

- **`预估时长分钟` 用粗略估算**（场景数 × 2 分钟）：这不是精确的制片排期数据，而是给创作者一个"这部剧本大概多长"的直觉。精确时长需要专业制片软件计算
- **单独成块**：统计信息与剧本内容正交，独立存储便于前端快速展示概览而不需要遍历整个场景列表

## 四、两阶段数据流与 Schema 的渐进构建

Schema 不是一次性生成的，而是在 Pipeline 的两个阶段逐步丰富：

### 阶段一：SceneAgent 输出（结构骨架）

```yaml
场景列表:
  - 场景编号: "s1"          # SceneAgent 生成
    来源章节: 1             # SceneAgent 生成
    场景标题:               # SceneAgent 生成
      地点: "..."
      时段: "早晨"
      场景细节: ""
    出场角色: ["c1"]        # SceneAgent 生成
    情绪基调: "宁静"         # SceneAgent 生成
    摘要: "林墨在花园里。"   # SceneAgent 生成（仅阶段一有，阶段二被 动作列表 替代）
    # 以下字段在此阶段均不存在：
    # 动作列表, 对白, 镜头建议, 转场方式
```

### 阶段二：ScriptAgent 输出（完整剧本）

```yaml
场景列表:
  - 场景编号: "s1"          # 沿用阶段一
    来源章节: 1             # 沿用阶段一
    场景标题:               # 沿用阶段一
      ...
    出场角色: ["c1"]        # 沿用阶段一
    情绪基调: "宁静"         # 沿用阶段一
    动作列表: [...]          # ScriptAgent 生成
    对白: [...]              # ScriptAgent 生成
    镜头建议: [...]          # ScriptAgent 生成
    转场方式: null           # ScriptAgent 生成
```

Pipeline 在 ScriptAgent 完成后将阶段二的完整结果回写到 State，后续 Validator 校验的就是包含了动作块和对白的完整场景。

**设计原因**：两阶段分离是因为它们的性质不同——场景切分（SceneAgent）是理解"什么发生了"，剧本生成（ScriptAgent）是理解"怎么呈现"。分开调用让每个 Agent 的 Prompt 更聚焦，同时给 HITL 提供了两个独立的编辑检查点。

## 五、校验规则

Validator 对剧本输出执行四个维度的校验：

### 5.1 Schema 结构校验

| 检查项 | 规则 |
|--------|------|
| 场景列表非空 | `场景列表` 不能为空数组 |
| 场景编号必填 | 每个场景必须有 `场景编号` |
| 场景标题完整 | `场景标题.地点` 和 `场景标题.时段` 必填 |
| 动作块非空 | `动作列表`（或兼容的 `动作描述`）不能为空 |

### 5.2 交叉引用校验

| 检查项 | 规则 |
|--------|------|
| 出场角色在角色表中 | 每个场景的 `出场角色` 中的 ID 必须存在于 `角色表` |
| 对白说话人在角色表中 | 每条对白的 `角色编号` 必须存在于 `角色表` |

当 `角色表` 为空时（尚未生成或 Scout 未运行），交叉引用校验自动跳过。

### 5.3 场景编号连续性

检查所有 `sN` 格式的编号是否从 1 开始连续递增。不连续时产生 Warning 而非 Error——编号断层不影响剧本可用性，但可能意味着 AI 遗漏了场景。

### 5.4 对白覆盖率

当场景数 > 3 时，检查有对白的场景占比是否 ≥ 30%。低于阈值时产生 Warning，提示可能遗漏了对话——纯动作场景占比过高在剧本改编中通常是异常信号。

## 六、设计权衡

### 为什么用中文字段名

代码中所有字段名统一使用中文（`场景列表`、`动作列表`、`对白`）。原因：

1. **LLM 输出质量**：Prompt 用中文描述 Schema，LLM 输出的 JSON key 也更倾向中文。强制翻译为英文 key 会在 LLM 侧引入额外的翻译错误
2. **编剧可读性**：最终 YAML 输出面向编剧，中文字段名比英文更可读
3. **一致性**：Pydantic 模型、LLM Prompt 模板、前端渲染、YAML 输出全部使用同一套中文字段名，不存在任何映射层

代价是代码中需要处理中文 key（如 `s.get("场景标题", {})`），但相比翻译错误和字段不一致带来的维护成本，这个代价是值得的。

### 为什么用 YAML 而非 JSON

| 维度 | JSON | YAML |
|------|------|------|
| 编剧可读性 | 差，花括号和大段转义 | 好，靠缩进表达结构 |
| 多行文本 | 差，需要 `\n` 或长字符串 | 好，原生多行支持 |
| 注释 | 不支持 | 支持 `#` 注释 |
| 影视习惯 | 不通用 | 与 Fountain/Final Draft 导出格式兼容 |

核心原因：这个工具的目标用户是作者和编剧，他们需要能肉眼读懂输出内容。JSON 对程序员友好，对创作者是灾难。YAML 在可读性和可解析性之间取得了平衡。

### 为什么动作块用对象数组而非字符串数组

对比两种设计：

```yaml
# 方案A：字符串数组（更简单）
动作列表:
  - "林墨走进房间。"
  - "他环顾四周。"

# 方案B：对象数组（当前设计）
动作列表:
  - 类型: "动作"
    内容: "林墨走进房间。"
  - 类型: "行为提示"
    内容: "（突然停住）"
```

选择方案B的原因：中文剧本创作中有"行为提示"（parenthetical）这个概念——它不是摄影机可见的动作，而是角色的微观心理或瞬时反应。它需要在排版上区别于标准动作描写。如果是纯字符串数组，这个区分就会丢失。

### 为什么不强制所有对白字段

`情绪`、`表演提示`、`行为提示` 都是可选字符串。强制填写会导致 LLM 生成"[无]"、"正常"等占位文本，污染输出。只标记真正重要的信息，让剧本保持简洁。

## 七、附录：完整 Pydantic 模型对照

以下为代码中实际使用的 Pydantic 模型（`backend/app/schemas/screenplay.py`），与 YAML Schema 一一对应：

| YAML 层级 | Pydantic 模型 | 关键字段 |
|-----------|--------------|----------|
| 根 | `Screenplay` | 格式版本, 元数据, 角色表, 场景列表, 改编说明, 统计信息 |
| 元数据 | `Metadata` | 剧本标题, 原著作者, 转换工具, 使用模型, 转换日期, 来源章节数 |
| 角色 | `Character` | id, name, aliases, description, relationships |
| 角色描述 | `CharacterDescription` | physical, personality, role |
| 角色关系 | `Relationship` | target_id, type, note |
| 场景 | `Scene` | 场景编号, 来源章节, 场景标题, 出场角色, 情绪基调, 动作列表, 对白, 镜头建议, 转场方式 |
| 场景标题 | `SceneHeading` | 地点, 时段, 场景细节 |
| 动作块 | `ActionBlock` | 类型 ("动作"\|"行为提示"), 内容 |
| 对白块 | `DialogueBlock` | 角色编号, 情绪, 表演提示, 台词内容, 行为提示 |
| 镜头建议 | `ShotSuggestion` | 镜头类型, 镜头描述, 可选 |
| 改编建议 | `AdaptationNote` | 场景编号, 来源章节, 分类, 严重程度, 内容 |
| 统计 | `Stats` | 场景数量, 角色数量, 对白总数, 预估时长分钟 |
